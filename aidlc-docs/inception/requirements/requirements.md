# PowerPal 要件分析書

**ドキュメント種別**: Requirements Specification (SRS)
**バージョン**: 1.1
**最終更新**: 2026-05-10
**ステータス**: Revised Draft (Hackathon MVP target)

---

## 1. 概要

### 1.1 プロダクトビジョン

PowerPal は、デスクワーカーの「自分の身体と注意力に関する小さな自己判断」を、デスクトップ常駐の電池キャラクターに肩代わりさせるプロダクトである。

一般的な健康管理サービスがユーザーに自律性を求めるのに対し、PowerPal はあえて自己管理を手放させる。ハッカソンテーマ「人をダメにする」への応答として、**外部の人間関係には一切踏み込まず、ユーザー本人の身体・注意力管理だけを越権的に肩代わりする**という境界を設計の中心に据える。

### 1.2 体験価値の一行表現

> 休む判断まで、手放そう。退勤時に、まだ少し余力を残すために。

### 1.3 主要差別化ポイント

| ポイント | 説明 |
|---|---|
| 予測的な予備充電 | 連続会議の前に、消耗を予測して先に休憩を提案する |
| 判断の肩代わり | 何をして休むかをユーザーに選ばせず、抽選で提示する |
| キャラクター性 | 分析的・冷静ではなく、感性的・任性的な文言で判定を伝える |
| 越権の境界 | ユーザー本人の身体には踏み込むが、職場の外部関係には一切介入しない |
| ローカルファースト | 個人データは端末内で完結させ、外部送信を極小化する |

## 2. スコープとフェーズ

本プロダクトは **Hackathon MVP Core** → **Hackathon Stretch** → **Phase 2** → **Phase 3** の段階的提供を想定する。

### 2.1 Hackathon MVP Core

**30 秒以内にコンセプトが伝わり、90 秒以内のデモで破綻せず、日常利用でも誤爆しにくいこと**を最優先する。

| ID | 機能 | 優先度 |
|---|---|---|
| MVP-01 | 電池キャラクター常駐表示（0-100%、5 色） | Core |
| MVP-02 | 朝の初期電池判定 + 感性的コメント | Core |
| MVP-03 | Google Calendar 連携（読み取り専用、mock fallback あり） | Core |
| MVP-04 | 予備充電モード（連続会議検知、15 分前通知） | Core |
| MVP-05 | 充電抽選（再抽選 1 日 3 回） | Core |
| MVP-06 | 隠身 / Quiet モード（高信頼シグナルの自動抑制 + 手動トグル） | Core |
| MVP-07 | 初回セットアップ（勤務時間・カレンダー接続・人前モード説明） | Core |

### 2.2 Hackathon Stretch

Core 完了後にのみ着手する。

| ID | 機能 | 優先度 |
|---|---|---|
| ST-01 | 退勤サマリー | Stretch |
| ST-02 | 強制充電モード（高閾値・非遮蔽型） | Stretch |
| ST-03 | 自動充電提案（連続作業後 / 会議後） | Stretch |

### 2.3 Phase 2: 製品化フェーズ

| 機能 | 概要 |
|---|---|

| Outlook Calendar / Microsoft Graph 連携 | 法人ユーザー向け |

| 詳細統計ダッシュボード | 日 / 週 / 月の振り返り |
| カスタム充電メニュー編集 UI | 追加 / 編集 / 削除 / 抽選対象切替 |
| 高精度な人前検知 | 画面共有・会議検知の OS 連携強化 |

### 2.4 Phase 3: 拡張フェーズ

| 機能 | 概要 |
|---|---|
| Apple Health / Fitbit 連携 | 心拍数・睡眠を初期電池判定に反映 |
| チーム機能（オプトイン） | 同じチームの平均電量を共有 |
| AI 推論強化 | 個人の消耗パターン学習 |
| Lottie アニメーション差し替え | キャラクター差別化 |

### 2.5 永久に対象外（プロダクト憲法によりスコープ外）

- 会議の自動キャンセル / 辞退
- 同僚・上司・顧客への自動返信・自動メッセージ
- カレンダー予定の自動変更・追加
- チャットステータス（Slack / Teams 等）の自動変更
- ユーザーの代わりに外部へ意思表示するあらゆる行為
- 企業向け勤怠監視機能

## 3. プロダクト原則（憲法）

### 3.1 基本原則

> **電池君はユーザー本人には越権できる。だが職場の外部関係には絶対に介入しない。**

### 3.2 越権が許される対象

ユーザー本人の：

- 目の疲れ
- 水分補給
- トイレのタイミング
- カフェイン補給
- 注意力
- 疲労感
- 休憩タイミング
- 今日の電池残量
- 退勤時の余力

### 3.3 介入してはならない対象

- 会議の自動キャンセル
- 同僚・上司・顧客への自動返信
- カレンダー予定の自動変更
- チャットステータスの自動変更
- ユーザーの代わりに外部へ意思表示する行為
- 画面共有中・プレゼン中・会議中など、他者に見える場面での強い割り込み

### 3.4 体験設計方針

- ユーザーに努力や改善を求めない
- 「自分の面倒を見る判断」を手放す体験を提供する
- ただし介入は段階的に行い、シグナルの確度より強い割り込みはしない
- 文言は分析的にせず、キャラクターの感性的判断として表現する
- 健康管理アプリ・HR 監視ツールに見えてはいけない

## 4. 想定ユーザー

### 4.1 主ターゲット

- デスクワーク中心の会社員
- 会議が多いプロジェクトメンバー
- 在宅勤務 / ハイブリッド勤務者
- 長時間 PC 作業を行うエンジニア・デザイナー・企画職
- 集中すると水分補給やトイレを忘れがちな人
- 健康管理アプリが続かなかった経験のある人

### 4.2 ユーザー課題

| 課題 ID | 内容 |
|---|---|
| UP-01 | 連続会議の前に水分補給・トイレを忘れる |
| UP-02 | 長時間画面を見続けて目が疲れる |
| UP-03 | 休憩すべきタイミングが分からない |
| UP-04 | 休憩したいが、何をすればいいか考えるのも面倒 |
| UP-05 | 退勤時には完全に疲れ切っている |
| UP-06 | 健康管理アプリが真面目すぎて継続できない |
| UP-07 | 自分の疲労状態を過小評価しがち |

## 5. 機能要件

### 5.1 電池モデル仕様

#### 5.1.1 電池残量

- **表現**: 0〜100% の整数値
- **位置づけ**: 生理学的な測定値ではなく、ユーザーの主観的余力を粗く代理するゲーム的メーター
- **表示**: iPhone 風電池アイコン + パーセンテージ
- **更新頻度**: アクティブ作業中は 1 分ごと / イベント発生時に即時再計算

#### 5.1.2 5 段階カラー

| 帯 | 状態名 | カラー | HEX | キャラクター表現 |
|---|---|---|---|---|
| 81-100% | 満タン | 鮮やかな緑 | `#34C759` | 元気いっぱい、跳ねる、明るい表情 |
| 61-80% | 元気 | 緑 | `#30D158` | 通常、笑顔 |
| 41-60% | 普通 | 黄 | `#FFCC00` | やや疲れ顔、口数減少 |
| 21-40% | 注意 | 橙 | `#FF9500` | 文句多め、揺れる、強い文言 |
| 0-20% | 危険 | 赤 | `#FF3B30` | ぐったり、眠そう、強い介入候補 |

#### 5.1.3 電池の消耗・回復

| 行動 | 影響 |
|---|---|
| 通常勤務 | **アクティブ作業 8 時間で概ね 100% を使い切る**粗い基準で、アクティブ分のみを毎分減算する |
| 連続作業（60 分超） | 基準消耗量の **1.1 倍** |
| 会議（60 分超） | 基準消耗量の **1.1 倍** |
| 水を飲む | +3% |
| 何もしない | +2% |
| 目を休める | +4% |
| トイレに行く | +5% |
| 立って伸びる | +5% |
| 屋外で深呼吸 | +5% |
| コーヒーを買う | +6% |
| 会社の下を散歩 | +8% |

補足:

- 消耗計算は **無操作やスリープ時間には原則適用しない**
- 朝の初期判定では天候補正も加味する。既定値は **雨 -3% / 雪 -5% / 晴・曇 0%**
- 回復量は「小休憩で少し戻る」程度に留め、数値より体験の整合性を優先する

### 5.2 機能要件一覧

#### FR-01: 電池キャラクター常駐表示

| 項目 | 内容 |
|---|---|
| 概要 | デスクトップ上に手足のある電池キャラクターを常駐表示する |
| 入力 | 現在の電池残量、最近のイベント |
| 出力 | キャラクター描画、表情、アニメーション |
| 詳細 | ドラッグで位置変更可能 / 小型表示・非表示切替 / 5 段階で表情とアニメーションが変化 |
| 優先度 | Core |

#### FR-02: 今日の初期電池残量判定

| 項目 | 内容 |
|---|---|
| 概要 | 勤務開始時に今日の初期電池残量を判定する |
| 入力 | 前日の退勤時刻、前日の最終電池、前日の充電回数、今日の会議総時間、今日の連続会議数、曜日、最近 7 日の傾向、天候 |
| 出力 | 初期電池残量（0-100%）、感性的コメント、手動修正欄 |
| 表示方針 | 分析的にせず、キャラクターの感性的判断として表現する |
| 優先度 | Core |

#### FR-03: 勤務・画面使用時間の計測

| 項目 | 内容 |
|---|---|
| 概要 | キーボード・マウス操作からアクティブ状態を判定し、連続作業時間を計測する |
| 詳細 | 既定の連続作業閾値は 60 分 / ユーザーが設定変更可 / アイドル判定は 5 分以上の無操作 |
| 優先度 | Core |

#### FR-04: Google Calendar 連携

| 項目 | 内容 |
|---|---|
| 概要 | Google Calendar から予定を読み取り、会議密度・連続会議を判定する |
| 認証 | OAuth 2.0（read-only スコープ `calendar.readonly`） |
| 取得対象 | プライマリカレンダー / ユーザー指定の追加カレンダー |
| 連続会議判定 | 参加者 2 名以上 + 間隔 10 分以内 + 合計 2 時間以上 |
| フォールバック | 認証失敗時 / オフライン時 / デモ時は mock JSON を使用 |
| 除外 | 終日予定は会議としてカウントしない / 自己のみ予定は会議としてカウントしない |
| 優先度 | Core |

#### FR-05: 予備充電モード

| 項目 | 内容 |
|---|---|
| 概要 | 高消耗な連続会議ブロックを検知した場合、開始前に充電行動を提案する |
| 触発 | 連続会議の開始 **15 分前** / 通知のスヌーズ可能 |
| 提案内容 | 水分補給、トイレ、コーヒー、目の休息、軽い散歩、深呼吸 |
| 表示 | 開始時刻・終了時刻・推奨メニュー・コメント・行動ボタン |
| 禁止 | 「1 日分を全自動化する」など、未定義の自動実行モードを MVP に入れない |
| 優先度 | Core |

#### FR-06: 充電抽選

| 項目 | 内容 |
|---|---|
| 概要 | ユーザーが「充電する」を押すと、サービス側が充電メニューを抽選して提示する |
| 詳細 | 主画面でメニューを直接選択しない / 1 日最大 3 回まで再抽選可能 / 上限後は自由入力にフォールバック |
| 優先度 | Core |

#### FR-07: 充電メニュー管理

| Phase 2 | 追加・編集・削除 UI を提供 |

| 水を飲む | 2 分 | +3% | 💧 |
| トイレに行く | 5 分 | +5% | 🚻 |
| 目を休める | 5 分 | +4% | 👁️ |

#### FR-08: 隠身 / Quiet モード

| 項目 | 内容 |
|---|---|
| 概要 | 他者に見える場面では、通知と表示強度を段階的に弱める |
| 高信頼シグナル | プレゼン用フルスクリーン / 既知会議アプリの前面化 / OS レベルのカメラ使用 / ユーザー手動の「人前モード」 |
| 低信頼シグナル | Chrome タブや Slack Huddle の詳細判定は MVP では自動化しない |
| 動作 | 大きなモーダルと強い割り込みを抑制し、必要なら 1 件だけ defer する |
| フォールバック | 判定に自信がない場合は「全通知停止」ではなく、「強い表示のみ抑制」へ倒す |

#### FR-09: 強制充電モード

| 項目 | 内容 |

| 優先度 | Stretch |

#### FR-10: 退勤サマリー

| 項目 | 内容 |
|---|---|
| 概要 | 退勤時に、その日の状態を電池君が短く振り返る |

| 優先度 | Stretch |

#### FR-11: 跨日記憶

| 項目 | 内容 |
|---|---|
| 概要 | 最近の利用傾向を記憶し、翌日以降のコメントや判定に反映する |

- 充電回数
- スキップ回数
- 連続会議数
- 低電量時間帯
- 使用された充電メニュー
- 再抽選回数

#### FR-12: 初回セットアップ

| 項目 | 内容 |
|---|---|
| 概要 | 初回起動時に最低限の設定と権限説明を行う |

## 6. 非機能要件

### NFR-01: 安全な介入範囲

ユーザー本人の身体・注意力・休憩行動のみを対象とする。外部の人間関係や業務予定への直接操作は実装しない。

### NFR-02: 他者に見える場面で邪魔しない

会議・画面共有・プレゼン・カメラ起動など、他者に見える可能性がある状態では、**強い通知・画面遮蔽・大きなアニメーションのみ**を抑制する。判定に自信がない場合は静音寄りにするが、プロダクト全体を無音化しない。

### NFR-03: ユーザーが最終的に解除できる

強めの介入であっても以下の操作が可能であること：

- 休憩開始
- 5 分後に再通知
- 今日だけ免提醒
- 設定から機能を弱める / 無効化

入力遅延などの操作妨害は実装しない。

### NFR-04: ローカルファースト

- 日々の履歴は SQLite にローカル保存
- カレンダー情報は判定にのみ利用し、予定詳細は永続化しない
- OAuth トークンは OS のセキュアストレージに保存
- Core MVP ではクラウド同期なし

### NFR-05: 通知編成

| 項目 | 目標値 |
|---|---|
| proactive 通知頻度 | 1 時間に最大 2 回 |
| 強い割り込み | 90 分に最大 1 回 |
| defer キュー | 隠身中に保持する通知は同時に 1 件まで |
| 重複通知 | 同一意図の通知は 30 分以内に再送しない |

### NFR-06: 軽量性

| 項目 | 目標値 |
|---|---|
| アイドル時 CPU 使用率 | 平均 1% 未満 |
| メモリ使用量 | 通常 200MB 以下 |
| 起動時間 | 3 秒以内 |

### NFR-07: キャラクター性

文言・アニメーション・反応は **キャラクター性を優先**する。ただし、感情表現が強くても判定根拠より強い割り込みにはしない。

### NFR-08: アクセシビリティ

- キーボードのみで全操作可能
- 強い介入 UI は ESC で即時解除可能
- 5 段階カラーは色のみでなく形状差でも区別可能にする

### NFR-09: プライバシー

- カレンダー予定のタイトル・参加者名はローカル DB に保存しない
- 画面共有検知のためにスクリーンショットを取得しない
- カメラ画像は一切取得しない

### NFR-10: 国際化

- MVP は日本語のみ
- Phase 2 で英語・中国語追加

### NFR-11: プラットフォーム

- **MVP: macOS**
- **Phase 2: Windows**
- Phase 3 以降で Linux を検討

## 8. リスクと対策

### R-01: 強い介入が邪魔すぎる

**リスク**: 業務中に強い割り込みが発生し、ユーザーが不快に感じる  
**対策**:

- Core MVP では強制充電を stretch 扱いにする
- 強い介入は複数シグナルがそろった時だけ発動
- 画面遮蔽型オーバーレイを使わない
- ユーザーが解除できる導線を必ず用意する

### R-02: 健康管理アプリに見えてしまう

**リスク**: 普通の生産性ツールに見える  
**対策**:

- 文言をキャラクター寄りにする
- 「管理」ではなく「判断を手放す」を前面に出す
- demo では朝の判定と予備充電を中心に見せる

### R-03: HR 監視ツールに見える

**リスク**: 勤務時間や会議密度を判定するため、会社に監視される印象を与える  
**対策**:

- データはローカル保存
- カレンダー詳細は保存しない
- 「評価」ではなく「相棒の感覚」として表現する

### R-04: 外部世界への介入リスク

**リスク**: 会議・チャット・予定変更に踏み込むと、現実の職場トラブルにつながる  
**対策**:

- プロダクト憲法で明文化し、実装上も書き込み系 API を一切呼ばない
- 自動返信・自動辞退・自動予定変更は永久にスコープ外

### R-05: 人前検知の誤検知 / 取りこぼし

**リスク**: 会議中に通知が出る、または逆に一日中黙る  
**対策**:

- Core MVP では高信頼シグナルしか自動化しない
- Chrome タブや Slack Huddle の詳細判定は手動トグルで補う
- 判定に自信がない場合は「強い UI だけ抑制」する

### R-06: Google Calendar 認証離脱

**リスク**: OAuth 同意で離脱され、MVP が機能しない  
**対策**:

- 認証スキップ用の mock データモードを用意する
- 認証なしでも朝判定・抽選・人前モードは使える状態を保証する
- 初回セットアップで read-only とローカル保存方針を明示する

## 10. 用語集

| 用語 | 定義 |
|---|---|
| 電池残量 | ユーザーの推定余力を表す粗いメーター |
| 充電 | 休憩・水分補給・トイレ・目の休息など、余力を回復するための小さな行動 |
| 予備充電 | 連続会議など高消耗イベント直前の予防的な充電提案 |
| 抽選 | ユーザーが「充電する」を押した際にメニューを自動選定する仕組み |
| 隠身モード | 他者に見える場面で大きな表示を抑制する状態 |
| Quiet モード | 判定に自信がない時に、強い表示だけを抑制する中間状態 |
| 跨日記憶 | 直近の利用傾向を翌日以降の判定や口調に反映する機能 |
| プロダクト憲法 | 機能設計の上位ルール。「ユーザー本人には越権するが、外部関係には介入しない」 |
| 連続会議 | 参加者 2 名以上 + 間隔 10 分以内 + 合計 2 時間以上の予定群 |
